---
name: sync-memory
description: Updates CLAUDE.md based on recent changes to the project. Run after adding new libraries, changing conventions, refactoring structure, or updating the design system. Keeps project context accurate without a full re-scan.
---

Update the existing `CLAUDE.md` to reflect recent changes in the project. Do not rewrite everything — surgically update only the sections that are outdated.

## Steps

### 1. Read the current CLAUDE.md
Understand what is already documented and when it was last updated.

If `CLAUDE.md` does not exist, tell the user to run `/map-project` first instead.

### 2. Identify what changed
Check recent git history to understand the scope of changes:
```bash
git log --oneline -20
git diff HEAD~10...HEAD --name-only
```

If the user described what changed (e.g. "I added Zustand", "I switched to Tailwind v4"), use that as the primary signal and skip broad diffing.

### 3. Re-scan only the affected areas

| If changed | Re-scan |
|---|---|
| New dependency added | `package.json`, usage in source files |
| New folder or module | Directory structure, conventions |
| Design system change | `tailwind.config.*`, CSS variables, theme files |
| New component pattern | 2-3 recent component files |
| New commands | `package.json` scripts, `Makefile` |
| Architecture change | Relevant source files for that layer |

### 4. Update CLAUDE.md

Edit only the sections that need updating. Preserve sections that are still accurate.

Add a comment at the top indicating the last update:
```markdown
> Generated by /map-project. Last updated by /sync-memory on <date>.
```

### 5. Report

List exactly which sections were updated and what changed. Example:
```
Updated:
- Stack: added Zustand 5.0 for global state
- Architecture Notes: documented store pattern used in src/store/
- Commands: added `npm run storybook`

Unchanged:
- Project Structure, Conventions, Design System
```

## When to run

Suggest to the user that `/sync-memory` is worth running after:
- Installing or removing a significant dependency
- Refactoring the folder structure
- Changing the design system or component library
- Establishing a new convention that should be followed going forward
- Adding new dev/build/test commands
