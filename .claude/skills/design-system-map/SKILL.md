Scan the frontend project and generate a `DESIGN_SYSTEM.md` file at the project root.

**Primary audience: AI models (Claude, v0, Lovable, Copilot, etc.) that will generate prototypes or new screens.**
The output must be precise enough that an AI can reproduce the visual system without guessing â€” exact values, real class names, actual prop names, and real code snippets from the codebase.

## Scanning steps

### 1. Identify the styling implementation

Determine the exact styling stack:
```bash
cat package.json | grep -E "tailwind|mui|chakra|antd|radix|shadcn|styled-components|emotion|stitches"
ls tailwind.config.* 2>/dev/null
find . -name "theme.*" -o -name "tokens.*" -o -name "variables.*" | grep -v node_modules | head -10
find . -name "globals.css" -o -name "global.css" -o -name "index.css" | grep -v node_modules | head -5
```

This determines how tokens should be referenced in the output:
- **Tailwind**: classes like `bg-primary`, `text-sm`, `rounded-lg`
- **CSS variables**: `var(--primary)`, `var(--radius)`
- **MUI/Chakra theme**: `theme.palette.primary.main`, `colorScheme.primary`
- **CSS-in-JS**: token object paths

### 2. Extract colors with implementation references

For each color, capture:
- The human name (Primary, Gray 500)
- The exact value (hex, HSL, RGB)
- **How to use it in code** (the actual class, variable, or token path)

Sources:
- `tailwind.config.*` â†’ `theme.colors` and `theme.extend.colors`
- `:root` CSS variables in `globals.css` or `variables.css`
- `theme.ts`, `tokens.ts`, `colors.ts`
- MUI `createTheme`, Chakra `extendTheme`

Extract: primary, secondary, accent, neutrals/grays, success, warning, error, info, backgrounds, surfaces, text colors, border colors. If dark mode exists, extract both.

### 3. Extract typography with implementation references

- Font families â†’ how loaded (Google Fonts import, local, CDN)
- Font size scale â†’ exact values AND the class/variable to use
- Font weights â†’ values AND classes
- Line heights, letter spacing if customized

### 4. Extract spacing, layout, and shape tokens

- Spacing scale (if customized beyond defaults)
- Breakpoints â†’ exact px values AND the class prefix
- Border radius scale â†’ values AND classes
- Shadow scale â†’ full CSS values AND classes
- Container max-widths

### 5. Scan components deeply

Find and fully read the most common UI components:
```bash
find src -type f \( -name "Button.*" -o -name "Input.*" -o -name "Card.*" -o -name "Badge.*" -o -name "Modal.*" -o -name "Dialog.*" -o -name "Select.*" -o -name "Textarea.*" -o -name "Avatar.*" -o -name "Alert.*" -o -name "Tooltip.*" -o -name "Dropdown.*" \) | grep -v node_modules | grep -v "\.test\." | grep -v "\.spec\." | grep -v "\.stories\." | head -20
```

For each component read, extract:
- All accepted props and their types
- All variant values (exact strings)
- All size values (exact strings)
- The class names or styles applied per variant
- A real usage example copied from the file or inferred from the props

### 6. Find real usage patterns

Look for pages or screens that compose multiple components together:
```bash
find src -type f \( -name "*.tsx" -o -name "*.jsx" \) | xargs grep -l "Button\|Card\|Form\|Modal" | grep -i "page\|screen\|view\|dashboard\|login\|home" | grep -v node_modules | head -10
```

Read 2-3 of these to extract real composition patterns (how components are assembled into layouts).

### 7. Icons and assets

- Icon library and import path
- How icons are used in code (size prop, className, etc.)
- Any image/asset conventions
